# 考勤异常检测与预警系统架构设计文档

## 1. 架构设计目标
系统面向中小企业考勤场景，采用轻量化、分层化、可扩展设计，重点支持复杂考勤异常的智能判定。

## 2. 总体架构
系统采用四层架构：
- 表现层
- 业务服务层
- 数据存储层
- 智能分析层

## 3. 分层设计

### 3.1 表现层
技术：`Vue3 + Element Plus`

职责：
- 登录页面
- 员工管理页面
- 打卡页面
- 异常与预警页面
- 报表页面
- 复核页面

### 3.2 业务服务层
技术：`SpringBoot + Spring Security`

职责：
- 用户与权限管理
- 考勤业务处理
- 异常检测调度
- 预警处理
- 报表统计
- 日志审计

### 3.3 数据存储层
技术：`MySQL + Redis`

职责：
- `MySQL`：使用 `system` 数据库持久化业务数据
- `Redis`：缓存会话、热点数据、统计结果、短期风险特征

核心数据表：
- `user`、`department`、`role`
- `device`、`faceFeature`、`rule`
- `attendanceRecord`、`attendanceException`、`exceptionAnalysis`
- `warningRecord`、`reviewRecord`、`operationLog`

命名约定：
- 表名使用简化风格，如 `attendanceRecord`、`warningRecord`
- 字段名使用小驼峰风格，如 `deptId`、`roleId`、`createTime`

### 3.4 智能分析层
组成：
- 人脸识别模块
- 规则引擎模块
- 大模型分析模块
- 融合决策模块

## 4. 核心架构思路
系统采用“规则引擎主判定 + 大模型辅助分析 + 人工复核兜底”机制：
- 基础异常由规则引擎判定
- 复杂异常由大模型基于多维上下文进行分析
- 低置信度结果转人工复核
- 复核结果反向支持策略优化

## 5. 复杂异常判定架构

### 5.1 判定流程
1. 采集打卡原始数据
2. 执行规则引擎初判
3. 提取风险特征
4. 调用大模型分析
5. 融合输出异常结果
6. 触发预警或复核

### 5.2 风险维度
- 身份维度
- 时间维度
- 设备维度
- 空间维度
- 行为维度
- 历史维度

## 6. 部署架构
建议采用轻量部署：
- 前端：浏览器访问
- 后端：Windows 本地部署
- MySQL：本地数据库，使用 `system` 库
- Redis：本地缓存服务
- 摄像头：本地设备
- 大模型：远程 API

## 7. 安全架构
- `Spring Security` 做认证授权
- 敏感数据加密存储
- 关键操作日志审计
- 接口访问鉴权
- 大模型调用做脱敏与审计

## 8. 性能设计
- `Redis` 缓存热点数据
- 数据库索引优化
- 复杂异常按需触发模型分析
- 异步处理统计与日志任务

## 9. 架构优势
- 低成本、易部署
- 兼顾规则稳定性和模型智能性
- 支持复杂异常扩展
- 适合论文设计与工程实现
