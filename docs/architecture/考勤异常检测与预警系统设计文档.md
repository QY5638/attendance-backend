# 基于大模型的复杂考勤异常判定考勤异常检测与预警系统设计文档

## 1. 系统设计概述

### 1.1 设计背景

随着企业数字化转型的不断深入，传统考勤管理方式在中小企业中逐渐暴露出代打卡识别困难、异常响应滞后、数据统计繁琐、管理成本较高等问题。尤其在复杂考勤场景下，仅依靠固定规则难以实现对异常行为的精准识别与有效预警。

针对上述问题，本系统在传统人脸识别考勤基础上，引入大模型 API 能力，对复杂考勤异常行为进行语义分析与综合判定，构建一套面向中小企业的低成本、易部署、可扩展、可解释的智能考勤异常检测与预警系统。

### 1.2 设计目标

本系统的总体目标是构建一套集员工管理、人脸打卡、基础异常检测、复杂异常智能判定、风险预警、统计分析于一体的考勤异常检测与预警平台。具体目标如下：

- 实现员工信息、人脸数据、部门信息和角色权限的统一管理
- 实现基于人脸识别的日常考勤打卡功能
- 实现迟到、早退、缺勤、重复打卡等基础异常的自动判定
- 基于大模型进一步开发复杂考勤异常判定功能，丰富判定维度与场景
- 整体优化系统的考勤异常判定逻辑，提升判定的精准性与实用性
- 支持风险预警、人工复核和结果归档，形成闭环管理机制
- 满足普通 Windows 办公电脑下的轻量化部署要求

### 1.3 设计原则

- 实用性原则：贴合中小企业真实考勤管理需求
- 准确性原则：融合规则引擎与大模型能力，提高异常判定准确率
- 低成本原则：采用轻量化技术栈与普通办公环境部署方案
- 安全性原则：加强敏感数据加密存储与访问控制
- 可扩展性原则：为后续异常场景拓展与模型优化预留接口
- 可解释性原则：异常结果应包含判定依据、风险等级和处理建议

## 2. 系统总体设计

### 2.1 系统定位

本系统面向中小企业考勤管理业务，定位为一套兼具基础考勤记录、异常识别、复杂行为分析、预警通知和统计报表功能的智能化管理平台。系统重点围绕“规则快速判定 + 大模型复杂分析 + 人工复核确认”展开设计，突出复杂考勤异常判定能力。

### 2.2 总体架构设计

系统采用前后端分离与分层架构设计，整体划分为表现层、业务服务层、数据存储层和智能分析层四个层次。

#### 2.2.1 表现层

表现层基于 Vue3 和 Element Plus 构建，负责界面展示和用户交互，主要包括：

- 系统登录界面
- 员工信息管理界面
- 人脸采集与打卡界面
- 异常记录查询界面
- 风险预警看板界面
- 统计报表与趋势分析界面
- 管理员人工复核界面

#### 2.2.2 业务服务层

业务服务层基于 SpringBoot 构建，负责系统核心业务逻辑，主要包括：

- 用户与权限管理服务
- 考勤记录管理服务
- 异常判定调度服务
- 预警通知服务
- 复核处理服务
- 报表统计服务
- 日志审计服务

#### 2.2.3 数据存储层

数据存储层采用 MySQL 与 Redis 组合实现：

- MySQL 使用 `system` 数据库，负责 `user`、`department`、`role`、`device`、`faceFeature`、`rule`、`attendanceRecord`、`attendanceException`、`exceptionAnalysis`、`warningRecord`、`reviewRecord`、`operationLog` 等业务数据的持久化
- Redis 负责会话缓存、热点统计缓存、风险特征缓存与临时分析数据存储

为提升数据库可读性与工程一致性，系统采用简化表名和小驼峰字段命名方式，例如 `attendanceRecord`、`warningRecord`、`deptId`、`createTime`。

#### 2.2.4 智能分析层

智能分析层是本系统的核心创新层，主要由以下部分组成：

- 人脸识别模块
- 规则引擎模块
- 大模型分析模块
- 提示词与策略管理模块
- 模型调用网关模块
- 融合决策模块
- 决策追踪与反馈学习模块

其中，规则引擎主要负责标准化异常的快速判定，大模型负责对规则难以准确覆盖的复杂异常进行上下文综合分析，提示词与策略管理模块负责按场景选择合适模板和版本，模型调用网关模块负责统一封装模型请求与输出解析，融合决策模块负责输出最终异常结论与风险等级，决策追踪与反馈学习模块负责记录规则结果、模型结果、最终决策以及人工复核反馈。

### 2.3 技术架构选型

- 前端框架：Vue3、Element Plus
- 后端框架：SpringBoot、Spring Security
- 数据库：MySQL
- 缓存中间件：Redis
- 人脸识别：OpenCV 或第三方人脸识别组件
- 智能分析：大模型 API
- 开发工具：IDEA、VSCode、Postman
- 部署环境：Windows 办公电脑或本地服务器

## 3. 系统功能模块设计

### 3.1 用户与权限管理模块

该模块主要完成基础用户和权限控制功能，包括：

- 员工信息增删改查
- 部门信息维护
- 管理员与员工角色配置
- 用户登录认证
- 权限校验与访问控制
- 用户状态启用与禁用

### 3.2 人脸采集与考勤打卡模块

该模块完成员工人脸录入、识别和打卡信息采集，具体功能包括：

- 摄像头实时采集图像
- 人脸检测与活体校验
- 人脸特征提取与匹配
- 打卡时间记录
- 打卡设备、IP、地点等信息采集
- 打卡结果反馈显示

### 3.3 基础异常判定模块

该模块采用规则引擎实现对常见考勤异常的快速识别，包括：

- 迟到判定
- 早退判定
- 缺勤判定
- 重复打卡判定
- 非法时间打卡判定

该模块强调实时性、稳定性和低成本，作为异常判定的第一层过滤机制。

### 3.4 复杂异常智能分析模块

该模块是系统创新重点，基于大模型对复杂场景进行综合分析，主要支持以下异常类型：

- 代打卡或替打卡
- 多设备异常打卡
- 多地点异常打卡
- 异常行为组合判定
- 持续性可疑打卡模式识别

系统将打卡记录、人脸比对结果、设备信息、地点信息、历史行为模式等多维特征进行结构化封装，再调用大模型执行复杂异常分析，输出异常结论、风险等级、判定依据、处理建议、相似案例摘要和置信度分值。

为增强模型能力的工程可控性，复杂异常智能分析模块还包括：

- 提示词模板选择与版本控制
- 结构化模型输出解析
- 决策追踪链路记录
- 基于人工复核结果的提示词与策略优化

### 3.5 风险预警模块

该模块负责对异常结果进行分级预警与消息提示，主要功能包括：

- 高风险异常实时预警
- 风险等级划分与标识
- AI 预警摘要生成
- 预警优先级排序
- AI 处置建议生成
- 预警列表展示
- 异常状态跟踪
- 预警信息归档

### 3.6 人工复核模块

人工复核模块用于弥补复杂异常场景中自动判定的不确定性，主要功能包括：

- 查看异常详情与模型分析依据
- 查看 AI 复核建议与相似案例摘要
- 管理员确认或驳回异常结果
- 记录误报和漏报情况
- 保存复核意见与处理记录
- 采集复核反馈标签，用于后续优化提示词与判定策略

### 3.7 统计分析模块

该模块负责对考勤数据和异常数据进行统计分析，具体包括：

- 个人考勤报表生成
- 部门考勤统计分析
- 出勤率与异常率统计
- 异常趋势分析
- 异常趋势智能解读
- 部门风险画像生成
- 周报和月报智能摘要生成
- 管理建议输出
- 报表导出与打印

### 3.8 智能决策支撑模块

该模块用于承载大模型全链路智能决策的公共底座能力，主要包括：

- 提示词模板管理
- 模型调用统一封装
- 结构化输出校验与解析
- 模型调用日志记录
- 决策链路追踪
- 人工反馈回流与策略优化

## 4. 复杂考勤异常判定设计

### 4.1 判定思路

为响应教师关于“进一步开发复杂考勤异常判定功能，丰富判定维度与场景，整体优化判定逻辑”的建议，系统采用“规则引擎 + 大模型协同判定”机制。

其中：

- 规则引擎用于处理结构清晰、标准明确的基础异常
- 大模型用于处理涉及多维信息、上下文关联和行为模式推理的复杂异常
- 融合模块用于对规则结果和模型结果进行统一决策

### 4.2 判定层次设计

系统异常判定逻辑划分为五层：

#### 第一层：打卡数据采集

采集员工身份、打卡时间、设备信息、IP 地址、地点信息、人脸相似度等原始数据。

#### 第二层：规则快速判定

依据预设考勤规则，对迟到、早退、缺勤、重复打卡等进行直接判断。

#### 第三层：风险特征提取

从单次打卡及历史记录中提取复杂异常分析所需特征，包括设备切换频率、地点变化、时间规律偏移、历史异常记录等。

#### 第四层：大模型智能分析

将结构化风险特征和场景上下文组织为标准化提示词，调用大模型进行复杂异常识别与解释生成。

#### 第五层：融合决策与预警输出

结合规则结果、大模型结果和置信度信息形成最终异常判断，并生成预警信息或复核任务。

### 4.3 判定维度设计

为提升异常识别的精准性与实用性，系统将复杂考勤异常判定从单一时间规则扩展到多维综合分析，主要包括：

- 身份维度：人脸相似度、身份匹配稳定性、注册信息一致性
- 时间维度：是否偏离正常上下班规律、是否集中在临界时间点
- 设备维度：设备是否频繁切换、设备是否可信
- 空间维度：地点是否异常、IP 是否异常、网络环境是否异常
- 行为维度：是否存在连续重复打卡、异常补卡、高频打卡尝试
- 历史维度：是否有多次异常历史、是否呈现持续性风险模式

### 4.4 重点复杂场景设计

#### 4.4.1 代打卡或替打卡

当员工打卡记录在表面上符合基本规则，但出现以下特征时，系统可判定为疑似代打卡：

- 人脸比对分数处于临界区间
- 打卡设备与历史常用设备明显不一致
- 打卡时间偏离个人历史规律
- 同一设备短时间服务多个不同员工
- 近期存在其他异常行为记录

大模型结合多维上下文信息进行综合推理，输出疑似代打卡结论及原因说明。

#### 4.4.2 多设备异常打卡

若同一员工在短时间内跨多个设备打卡，且设备变更缺乏合理性，则系统将提升该记录的风险等级，并结合大模型分析是否存在异常打卡可能。

#### 4.4.3 多地点异常打卡

若员工在短时间内出现在不合理的空间跨度地点进行打卡，系统将结合打卡时间间隔、地点信息与历史轨迹进行综合分析。

#### 4.4.4 异常行为组合判定

当单次记录无法直接判为异常，但多条记录组合呈现明显异常模式时，系统将触发组合行为分析。例如：

- 连续多日在上班截止前临界时间打卡
- 多次补卡申请伴随设备切换
- 同一时间段出现多次打卡尝试
- 多日高频异常但单条记录均未超规则阈值

该场景由大模型更适合完成行为模式归纳和风险解释。

### 4.5 全链路智能决策增强设计

为进一步突出大模型在系统中的核心作用，本系统不将大模型局限于复杂异常识别，而是将其扩展到异常识别后的关键业务链路中，形成“识别 - 预警 - 复核 - 统计解释 - 反馈学习”的全链路智能决策机制。

在异常识别阶段，大模型负责对多维风险特征进行综合推理，并输出结构化结论、置信度、判定依据和处理建议；在预警阶段，大模型基于异常结果进一步生成预警摘要、优先级和建议处置动作；在人工复核阶段，大模型提供证据归纳、相似案例与复核建议，辅助管理员提升判断效率；在统计分析阶段，大模型对趋势数据、部门风险分布和历史复核结果进行归纳，输出周报、月报与管理建议。

同时，系统保留规则引擎和人工复核兜底机制，避免完全依赖大模型造成成本和稳定性风险。人工复核的确认、驳回、误报和漏报反馈会被记录，并反向作用于提示词版本、特征权重和融合决策策略，从而形成可持续优化的智能闭环。

## 5. 系统安全与性能设计

### 5.1 安全设计

为保证员工隐私和考勤数据安全，系统采取如下安全措施：

- 用户登录采用身份认证和权限控制机制
- 敏感信息如人脸特征、联系方式等进行加密存储
- 管理员和员工角色进行分级访问控制
- 关键操作保留操作日志和审计记录
- 大模型调用过程进行访问校验和异常日志记录
- 避免敏感原始数据在模型调用中直接暴露
- 模型输入前进行字段脱敏与摘要化处理
- 模型输出结果进行结构化校验与安全审查
- 决策链、模型调用日志和人工反馈结果保留审计轨迹

### 5.2 性能设计

结合开题报告要求，系统主要性能指标如下：

- 核心功能响应时间不超过 3 秒
- 支持 30 人并发访问
- 异常识别准确率达到 95% 以上
- 支持普通 Windows 办公环境下稳定部署运行

为满足上述要求，系统采用以下优化策略：

- Redis 缓存热点数据与统计结果
- 对考勤记录、异常记录建立数据库索引
- 基础异常优先走规则引擎，减少大模型调用次数
- 将统计任务和日志处理设计为异步执行
- 对复杂异常采用按需触发分析机制，降低系统资源消耗
- 对提示词模板和高频策略结果进行缓存，减少重复构造成本
- 当模型调用失败或响应超时时，自动降级为规则判定或转人工复核

## 6. 系统创新点

- 引入规则引擎与大模型协同的复杂考勤异常判定机制
- 将异常判定从单一规则扩展到多维上下文综合分析
- 增强代打卡、多设备异常、多地点异常和行为组合异常的识别能力
- 增加异常结果解释与人工复核闭环，提升结果可用性与可信度
- 在低成本部署前提下实现考勤管理智能化升级，适配中小企业应用场景
- 将大模型从单点复杂异常判定扩展为全链路智能决策能力
- 建立提示词模板、模型日志、决策追踪和反馈学习的工程化闭环
- 让异常识别、预警、复核和统计解释均具备可解释的 AI 辅助能力

## 7. 详细设计说明

### 7.1 系统角色设计

系统主要包括管理员和员工两类角色：

#### 7.1.1 管理员

管理员主要负责：

- 员工和部门信息管理
- 考勤规则配置
- 异常记录查看
- 风险预警处理
- 人工复核操作
- 报表导出与数据分析

#### 7.1.2 员工

员工主要负责：

- 登录系统
- 完成人脸录入
- 日常打卡
- 查看个人考勤记录
- 提交补卡申请

### 7.2 功能结构设计

系统划分为以下子系统：

- 用户与权限管理子系统
- 人脸采集与识别子系统
- 考勤记录管理子系统
- 基础异常判定子系统
- 复杂异常智能分析子系统
- 风险预警与人工复核子系统
- 统计分析与报表子系统
- 智能决策支撑子系统

### 7.3 数据库设计

本系统关系型数据库命名为 `system`，表名采用简化风格，字段采用小驼峰命名方式，以提升可读性并与后端对象命名保持一致。

#### 7.3.1 核心数据表设计

1. `user` 用户表

- id：主键
- username：登录账号
- password：登录密码
- realName：姓名
- gender：性别
- phone：联系电话
- deptId：所属部门
- roleId：角色编号
- status：用户状态
- createTime：创建时间

2. `department` 部门表

- id：主键
- name：部门名称
- description：部门描述

3. `role` 角色表

- id：主键

### 7.5 关键业务流程设计

#### 7.5.1 正常打卡流程

1. 员工进入打卡页面
2. 摄像头采集人脸图像
3. 系统执行人脸识别和身份校验
4. 生成打卡记录并保存原始特征信息
5. 规则引擎进行基础异常检查
6. 若无异常，则写入正常打卡记录

#### 7.5.2 复杂异常分析流程

1. 员工完成打卡
2. 系统执行规则引擎初步判断
3. 若命中风险特征，则提取时间、设备、地点、历史记录等信息
4. 组织结构化提示词并调用大模型 API
5. 获取复杂异常分析结果与风险等级
6. 融合模块输出最终判定
7. 异常结果落库并触发预警
8. 若结果不确定，则转人工复核

#### 7.5.3 人工复核流程

1. 管理员查看预警信息
2. 进入异常详情页面
3. 查看模型分析依据和历史记录
4. 进行确认、驳回或标注误报处理
5. 系统更新异常状态并保存复核记录

#### 7.5.4 智能预警与反馈学习流程

1. 异常判定完成后生成结构化 AI 分析结果
2. 系统根据异常风险和模型建议生成预警摘要与优先级
3. 管理员在复核页面查看 AI 建议、相似案例和决策链
4. 管理员提交确认、驳回、误报或漏报反馈
5. 系统将反馈标签写入复核记录，并作为后续提示词和判定策略优化依据

### 7.6 判定策略设计

#### 7.6.1 基础异常判定策略

基础异常由规则引擎快速判断：

- 超过规定上班时间阈值判定为迟到
- 早于规定下班时间阈值判定为早退
- 无打卡记录判定为缺勤
- 短时间多次提交打卡判定为重复打卡

#### 7.6.2 复杂异常判定策略

复杂异常由风险特征提取与大模型联合完成：

- 若同一设备在短时间内被多个员工高频使用，则提升代打卡风险
- 若同一员工在短时间内跨设备、跨地点打卡，则提升异常风险
- 若存在边界时间高频打卡、频繁补卡和重复尝试等组合行为，则触发组合异常分析
- 若分析结果不稳定，则输出疑似异常并建议人工复核

#### 7.6.3 融合决策策略

- 当规则结果明确时，以规则结果为主
- 当规则结果不充分但模型风险高时，标记为复杂异常
- 当模型置信度不足时，转人工复核
- 人工复核结果可反向用于优化提示词和判定策略
- 当模型输出高风险且处置建议强烈时，预警模块提升优先级并建议立即处理
- 当历史复核持续驳回某类模型判断时，降低对应提示词策略权重并触发模板优化

### 7.7 部署设计

系统采用轻量化部署方案，适配普通 Windows 办公电脑，具体方式如下：

- 前端页面通过浏览器访问
- SpringBoot 后端服务本地部署
- MySQL 和 Redis 本地安装运行
- 摄像头通过浏览器或本地设备接入
- 大模型通过外部 API 调用

该方案有利于降低系统部署成本和维护难度，符合中小企业的实际应用需求。

## 8. 结论

本系统围绕中小企业考勤管理需求，结合 SpringBoot、Vue3、人脸识别和大模型 API 技术，设计了一套兼具基础异常快速识别、复杂异常智能分析、智能预警辅助、复核建议生成和统计解释能力的考勤异常检测与预警系统。系统在总体架构、功能模块、异常判定逻辑、数据库结构和接口设计等方面形成了较为完整的设计方案，尤其在大模型全链路智能决策能力、可解释性增强和反馈学习闭环方面，进一步强化了系统的创新性与工程完整度。

通过规则引擎、大模型和人工复核协同机制，系统不仅能够实现传统考勤异常识别，还能够面向代打卡、多设备异常打卡、多地点异常打卡和组合行为异常等复杂场景提供更精准、更实用、更可解释的分析能力，并将模型能力延伸到预警、复核和统计解释环节，为后续系统实现、论文撰写和答辩展示奠定了更具亮点的设计基础。
