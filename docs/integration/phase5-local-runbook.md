# Phase 5 本地联调手册

## 1. 适用范围

- 本手册只覆盖前后端本地启动与最小主链联调顺序，不覆盖部署、CI、全量回归或最终验收清单。
- 最小主链基线限定为：`登录 -> 异常查询 -> 预警查询 -> 复核 -> 统计查询`。
- `实时新打卡` 只作为带前置条件的独立链路，不属于首包稳定基线。
- 内容分别与后端 README 的配置/启动说明、前端 README 的环境变量/代理/dev 启动说明保持一致；如果两边配置未对齐，先以 README 中的要求修正本地环境，再继续联调。
- LLM、地图、多地点异常只属于可选增强链路，不作为本手册最小主链的基线前提。

## 2. 前置依赖

### 2.1 后端依赖

- JDK 8。
- Maven 3.x，并确保 `mvn -v` 使用的是 Java 8。
- 可访问的 MySQL 实例。
- 可访问的 Redis 实例。
- 如需验证复杂异常分析、地图展示或多地点异常，再额外准备可用的 LLM 服务配置与地图服务配置；这些不属于最小主链必填项。

### 2.2 前端依赖

- Node.js `18` 及以上版本。
- npm。

### 2.3 最小数据条件

- 至少准备一个可登录账号。
- 推荐直接导入仓库自带的 `schema + seed` 数据，先复用现成异常、预警、复核样本完成最小主链。
- 如果要额外验证实时新打卡，再准备 `EMPLOYEE` 账号、已完成人脸采集的结果、可提交的 `imageData` 和可用于打卡的设备数据。
- 如果本地数据库不是按仓库当前 `schema + seed` 初始化的，需要自行准备等价的异常、预警、复核数据；本手册不编造额外 SQL 或手工补数步骤。

## 3. 推荐启动顺序

### 3.1 后端配置

- 在后端仓库根目录把 `application-local.example.yml` 复制为 `application-local.yml`。
- `application-local.yml` 必须与 `pom.xml` 同级，不能放进 `src/main/resources`。
- 至少补齐以下配置：`spring.datasource.url`、`spring.datasource.username`、`spring.datasource.password`、`spring.redis.host`、`spring.redis.port`。
- `spring.redis.password` 也要显式处理：Redis 开启密码时填写真实值；如果本地 Redis 无密码，需要在 `application-local.yml` 中显式设为 `~`（YAML null）或其他等价空值，不能删除该键后回落到默认 `${REDIS_PASSWORD:change_me}`。
- 如果要验证复杂异常分析或地图链路，再补齐 `app.llm.provider`、`app.llm.base-url`、`app.llm.model`、`app.llm.api-key`、`app.map.provider`、`app.map.base-url`、`app.map.api-key`。
- 若按仓库 SQL 脚本原样初始化数据库，数据源库名要指向 `system`；当前默认回退库名已对齐为 `system`，如显式覆盖 `DB_URL` 也应保持一致。
- 排查配置问题时，不要只看 `application-local.yml`，还要同时核对 `src/main/resources/application.yml` 与 `src/main/resources/application-dev.yml`。

### 3.2 后端启动

- 先启动 MySQL 与 Redis。
- 按顺序执行数据库脚本：`sql/schema/数据库建表SQL.sql`，再执行 `sql/seed/测试数据插入SQL.sql`。
- 在后端仓库根目录执行 `mvn spring-boot:run`。
- 默认端口为 `8080`。
- 必须从仓库根目录启动，确保 `./application-local.yml` 能被正确加载。

### 3.3 前端配置

- 在前端仓库根目录把 `.env.example` 复制为 `.env.local`。
- 最小推荐值：`VITE_API_BASE_URL=/api`、`VITE_API_PROXY_TARGET=http://127.0.0.1:8080`。
- 如果要验证地图展示或地图相关异常，再补齐有效的 `VITE_AMAP_KEY`。
- 修改 `.env.local` 后，需要重新启动前端开发服务器才能生效。

### 3.4 前端启动

- 首次启动前执行 `npm install`。
- 在前端仓库根目录执行 `npm run dev`。
- 项目未额外指定端口时，Vite 默认使用 `5173`。
- 本地代理只在 `npm run dev` 的开发服务器中生效。

## 4. 联调前推荐数据准备方式

### 4.1 最小基线：优先复用仓库 seed 数据

- 先按后端 README 顺序导入 `sql/schema/数据库建表SQL.sql` 和 `sql/seed/测试数据插入SQL.sql`，并确保应用连接的是 `system` 库。
- 当前 seed 已提供一组可直接用于最小联调的样本数据：
  - 登录：仓库现有测试与测试文档均使用 `admin/123456` 作为管理员登录样例。
  - 规则异常：`3002` 为 `LATE`（李四，`2026-03-26 09:16:00`），`3003` 为 `REPEAT_CHECK`（张三，`2026-03-26 09:00:00`）。
  - 对应预警：`5002`、`5003` 初始状态均为 `UNPROCESSED`，适合继续走“去复核 -> 提交复核 -> 回查预警状态”。
  - 已有复核样本：`3001 -> 5001 -> 6001` 已是一条完成过复核的历史记录，适合只读查看，不必作为最小主链的必经依赖。
- 推荐执行方式：
  1. 先使用 `admin/123456` 完成 `ADMIN` 侧登录，只把它作为异常、预警、复核、统计页面的稳定入口。
  2. 异常、预警、复核、统计这四步优先复用 seed 中的规则型样本 `3002/5002` 或 `3003/5003`，不要把实时新打卡当成最小联调前置条件。
  3. 在复核页对 `3002` 或 `3003` 提交一次复核后，再回到预警页确认对应预警状态从 `UNPROCESSED` 变为 `PROCESSED`。
- 如果你导入的不是当前仓库这份 seed，或库内已经混入历史脏数据，上述 ID、用户名、状态可能对不上；此时应先重建 `system` 库再联调。

### 4.2 可选独立链路：实时新打卡

- 这条链路不属于首包稳定基线，应与最小主链分开验证。
- 前置条件：准备可登录的 `EMPLOYEE` 账号、已完成人脸采集的结果、可提交的 `imageData`，以及可用于打卡的设备数据。
- 原因：前端打卡页要求 `imageData`，后端打卡只有在人脸校验成功后才会写入；现有 seed 中的 `faceFeature.featureData` 与本地 MD5 比对实现并不天然对应，不能把 `admin/123456` 或 seed 样本直接当成一次稳定的新打卡样例。
- 如果要验证这条链路，建议先单独完成“员工登录 -> 人脸采集 -> 打卡提交”，再回到 seed 主链继续验证异常、预警、复核、统计。

### 4.3 其他可选增强链路

- 地图展示：需要前端 `.env.local` 中提供有效 `VITE_AMAP_KEY`。
- 复杂异常分析：需要后端补齐 `app.llm.*` 配置。
- 多地点异常：需要地图/空间能力可用，并准备带坐标的设备或历史打卡数据。
- 上述三项只建议作为扩展验证记录，不应阻塞本手册的最小主链联调。

## 5. 最小联调路径

- 下表只覆盖 seed 数据优先的稳定主链，不包含现场实时新打卡；如需验证实时新打卡，请单独按 `4.2` 的前置条件执行。

| 步骤 | 入口页面 | 依赖条件 | 操作动作 | 预期结果 |
| --- | --- | --- | --- | --- |
| 登录 | 浏览器打开前端开发地址后的登录页 | 后端已启动；前端代理正常；已导入当前 `schema + seed` 或已准备等价账号 | 优先使用 `admin/123456` 登录 | 登录成功并进入业务页面；后续接口请求不再因未认证失败 |
| 异常查询 | 异常查询页 | 已导入 seed，或本地已有可查询异常数据 | 优先查询 seed 中的规则异常 `3002` 或 `3003`；若页面默认按时间排序，也可直接查看最新异常列表 | 能看到异常记录；最小主链下至少能查到规则型异常，不依赖 LLM、地图或多地点场景 |
| 预警查询 | 预警查询页 | 已存在与异常关联的预警数据 | 查询 `3002/3003` 对应的 `5002/5003`，或查看最新预警列表 | 能看到与异常关联的预警记录，且初始状态为待处理 |
| 复核 | 异常页详情弹窗的“去复核”入口，或预警页列表的“去复核”入口 | 异常页或预警页列表已能加载 `3002` 或 `3003` 对应数据 | 从异常页或预警页跳转到复核页，并对 `3002` 或 `3003` 提交一次复核 | 复核页可基于 `exceptionId` 正常初始化；提交成功后，关联预警状态应更新为 `PROCESSED` |
| 统计查询 | 统计页 | 登录、异常、预警、复核数据已落库 | 查询当前统计结果，重点查看异常汇总是否包含已导入或已提交复核的异常数据 | 统计结果能返回当前库内已有的异常汇总结果，不要求最小主链必须覆盖地图、复杂异常或多地点计数 |

## 6. 最小验证命令与人工验证动作

### 6.1 后端最小命令

```bash
mvn spring-boot:run
mvn "-Dtest=DeviceManagementIntegrationTest,AuthSecurityIntegrationTest,ModuleSkeletonBeansTest" test
mvn -DskipTests compile
```

### 6.2 前端最小命令

```bash
npm install
npm run dev
npm test
npm run build
```

### 6.3 人工验证动作

1. 先确认后端启动日志无数据源、Redis、端口占用类错误，且服务监听在 `8080`。
2. 打开前端开发地址，确认页面可加载且接口请求通过 `/api` 代理到后端。
3. 如果本地库按当前 `schema + seed` 重建，优先使用 `admin/123456` 登录。
4. 登录后先进入异常页和预警页，优先定位 seed 中的规则型样本 `3002/5002` 或 `3003/5003`。
5. 通过“去复核”入口进入复核页，对其中一条待处理记录提交一次复核。
6. 回到预警页确认对应记录状态已变化，再到统计页确认能查到当前库内异常汇总结果。
7. 只有在你明确要验证实时新打卡时，才额外准备 `EMPLOYEE` 账号、人脸采集结果和可用 `imageData`，单独验证“员工登录 -> 人脸采集 -> 打卡提交”链路。
8. 只有在你明确要做其他扩展验证时，才额外检查地图展示、复杂异常分析或多地点异常链路。

## 7. 常见联调阻塞排查顺序

1. 先检查后端本地配置文件位置是否正确：`application-local.yml` 必须在后端仓库根目录。
2. 再检查后端配置叠加是否正确：同时核对 `application.yml`、`application-dev.yml`、`application-local.yml`。
3. 检查 MySQL、Redis 是否已启动，数据源库名、账号密码、Redis 地址端口是否正确。
4. 检查数据库初始化顺序是否正确：必须先建表，再导入测试数据。
5. 检查后端是否真的从仓库根目录启动，且监听在 `8080`。
6. 检查前端 `.env.local` 是否存在，`VITE_API_BASE_URL` 是否仍为 `/api`，`VITE_API_PROXY_TARGET` 是否指向 `http://127.0.0.1:8080`。
7. 修改 `.env.local` 后如果问题仍在，先重启 `npm run dev`，再重试接口。
8. 如果按当前 seed 初始化后 `admin/123456` 仍无法登录，先检查是否真的重建了 `system` 库并重新导入了当前仓库的 seed。
9. 如果你额外验证实时新打卡时失败，先检查是否使用了 `EMPLOYEE` 账号、是否已完成人脸采集并提供可用 `imageData`；这条链路不属于最小主链，先切回 seed 中现成的 `3002/5002` 或 `3003/5003` 继续验证异常、预警、复核、统计链路。
10. 如果异常页或预警页查不到上述样本，优先排查数据库是否混入旧数据或没有按当前 `schema + seed` 初始化。
11. 如果地图不显示、复杂异常分析失败或多地点异常未命中，按扩展验证处理，分别检查 `VITE_AMAP_KEY`、后端 `app.llm.*`、地图配置与坐标数据，不要和最小主链问题混在一起排查。

## 8. 使用边界

- 本手册用于本地联调起步，不替代最终验收清单。
- 若需要补充部署、导出、模型日志、系统配置或全量回归内容，应另写对应文档，不要继续扩写本手册。
