# BE-01 用户与权限基础模块开发指南

## 1. 文档说明

- 本文件用于指导 `BE-01` 的独立开发。
- 开发前先读取：
  - `D:\Graduation project\backend\AGENTS.md`
  - `D:\Graduation project\backend\docs\progress\后端并发开发模块清单.md`
  - `D:\Graduation project\backend\docs\api\API接口设计文档.md`

## 2. 模块目标

- 完成登录、用户管理、部门管理、角色管理和最小权限边界。
- 为前端认证壳层和后续业务模块提供稳定的用户主体与鉴权基础。

## 3. 模块边界

- 只处理：`auth`、`user`、`department`、`role`。
- 不处理：`device`、`face`、`attendance`、`exceptiondetect`、`warning`、`review`、`statistics`。

## 4. 前置依赖

- 无业务前置依赖。
- 需兼容现有统一返回、全局异常处理、安全配置骨架。

## 5. 固定接口清单

1. `POST /api/auth/login`
2. `GET /api/user/list`
3. `POST /api/user/add`
4. `PUT /api/user/update`
5. `DELETE /api/user/{id}`
6. `GET /api/department/list`
7. `POST /api/department/add`
8. `PUT /api/department/update`
9. `DELETE /api/department/{id}`
10. `GET /api/role/list`
11. `POST /api/role/add`
12. `PUT /api/role/update`
13. `DELETE /api/role/{id}`

## 6. 固定数据表与字段约定

- 主表：`user`、`department`、`role`。
- 用户关键字段：`username`、`realName`、`deptId`、`roleId`、`status`、`createTime`。
- 登录返回字段：`token`、`roleCode`。
- 管理接口权限边界：管理员可访问，员工不可访问。

## 7. 代码目录范围

- `src/main/java/com/quyong/attendance/module/auth/`
- `src/main/java/com/quyong/attendance/module/user/`
- `src/main/java/com/quyong/attendance/module/department/`
- `src/main/java/com/quyong/attendance/module/role/`
- `src/main/java/com/quyong/attendance/config/`
- `src/main/java/com/quyong/attendance/common/`
- `src/test/java/com/quyong/attendance/`
- `src/test/resources/`

## 8. 详细开发步骤

1. 建立测试环境最小数据源和 `user/department/role` 种子数据。
2. 实现登录 DTO、VO、Controller、Service 和最小 token 方案。
3. 配置安全放行 `/api/auth/login` 与 `/api/health`。
4. 实现未登录统一 `401` 返回。
5. 实现部门列表、新增、修改、删除。
6. 实现角色列表、新增、修改、删除。
7. 实现用户列表、新增、修改、删除。
8. 补齐用户名唯一、角色编码唯一、部门角色引用合法性校验。
9. 实现管理员与员工的最小权限边界，越权返回 `403`。
10. 补齐登录、CRUD、权限边界测试。
11. 若接口字段微调，同步 API 和测试文档。

## 9. 最小验证命令

```bash
mvn -Dtest=AttendanceApplicationTests,AuthControllerTest test
mvn test
```

## 10. 完成标准

- 登录成功/失败可用。
- 用户、部门、角色 CRUD 可用。
- 未登录返回 `401`，越权返回 `403`。
- 字段与 API 文档一致。

## 11. 禁止改动项

- 不引入复杂 RBAC。
- 不顺手改设备、人脸、打卡等其他模块。
- 不擅自变更数据库核心命名风格。

## 12. 文档同步要求

- 改接口时同步 `docs/api/API接口设计文档.md`。
- 改测试范围时同步 `docs/test/测试用例文档.md`。
- 若改库结构，必须同步 SQL 和数据库文档。
