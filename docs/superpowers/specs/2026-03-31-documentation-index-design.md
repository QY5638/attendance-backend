# 项目文档索引设计说明

## 1. 目标

在 `D:\Graduation project\backend\docs` 下新增统一入口文档 `README.md`，让后续开发、联调、文档维护和论文整理都能从一个页面快速找到对应资料，并明确每份文档的用途、阅读顺序与同步规则。

## 2. 设计范围

本次设计仅覆盖项目内文档导航与引用一致性，不涉及业务代码、接口设计或数据库结构调整。

包含内容：

- 新增 `D:\Graduation project\backend\docs\README.md` 作为文档工作台首页。
- 在索引中整理现有目录：`requirements`、`architecture`、`database`、`api`、`test`、`reference`。
- 在索引中补充 `sql\schema` 与 `sql\seed` 的使用说明。
- 修正文档内仍残留的旧外部路径，保证项目内路径自洽。

不包含内容：

- 不调整现有文档文件名。
- 不重写已有需求、架构、数据库、接口、测试正文。
- 不引入新的文档管理工具或自动化脚本。

## 3. 文件设计

### 3.1 新增文件

- `D:\Graduation project\backend\docs\README.md`

职责：作为项目文档统一入口，提供分类索引、推荐阅读顺序、同步规则和 SQL 入口说明。

### 3.2 修改文件

- `D:\Graduation project\backend\docs\test\测试用例文档.md`

职责：将仍指向旧外部目录的测试数据导入路径改为项目内路径，确保 README 与正文引用一致。

## 4. README 结构设计

`D:\Graduation project\backend\docs\README.md` 采用“工作台版”结构，按以下顺序组织：

1. 文档说明：说明该文件是项目文档总入口，适用于开发、测试、答辩准备。
2. 推荐阅读顺序：从需求到架构，再到数据库、接口、测试和 SQL，形成一致阅读路径。
3. 分类索引：按目录列出每份文档，并说明“解决什么问题”“什么时候看”。
4. 文档同步规则：明确数据库、接口、测试、模块边界变更时必须同步的文件。
5. SQL 入口：单独列出建表脚本和测试数据脚本路径，便于初始化环境。

## 5. 内容风格约束

- 全文使用中文。
- 以简洁、可导航、可执行为目标，避免空泛描述。
- 路径采用项目内绝对路径或清晰的相对定位，不再引用旧外部资料目录。
- 说明文字以“做什么、何时看、改动时同步什么”为核心。

## 6. 已识别的一致性问题

当前文档中发现一处旧路径残留：

- `D:\Graduation project\backend\docs\test\测试用例文档.md` 中测试数据脚本仍指向外部目录。

本次实现需将其改为：

- `D:\Graduation project\backend\sql\seed\测试数据插入SQL.sql`

## 7. 实施原则

- 保持最小必要改动。
- 不打乱现有目录结构。
- 先保证项目内路径闭环，再考虑后续扩展。
- 索引文档应能直接支撑后续模块开发与论文资料查找。

## 8. 验证方式

完成实现后，至少执行以下检查：

- 确认 `D:\Graduation project\backend\docs\README.md` 已创建。
- 确认 README 中列出的核心文档路径均实际存在。
- 检索项目内文档，确认不再残留旧外部资料目录引用。
- 核对 `D:\Graduation project\backend\docs\test\测试用例文档.md` 中测试数据脚本路径已更新为项目内路径。

## 9. 预期结果

实现完成后，项目文档将具备以下特征：

- 新成员可通过 `D:\Graduation project\backend\docs\README.md` 快速理解资料结构。
- 开发过程中可根据 README 判断需求、架构、数据库、接口、测试文档的查阅入口。
- 文档更新责任更清晰，减少只改代码不改文档或只改 SQL 不改说明的情况。
- 项目文档体系更适合后续论文撰写与答辩展示。
