# Phase 4 Qwen 真实接入可观察性设计

## 背景

当前 `Phase 4 - 智能链路治理完善` 要先锁定一条最小治理闭环。现有后端已经具备：

- `QwenModelGateway` 的真实调用能力。
- `ExceptionAnalysisOrchestratorImpl.complexCheck()` 的成功日志落库。
- Mock/失败降级路径的日志与决策追踪落库。

但成功日志当前只记录通用业务信息，缺少对“本次成功调用来自 Qwen 真实接入”的直接可观察证据，不利于满足“真实配置下能看到模型调用记录”的阶段完成定义。

## 目标

只补齐一条最小闭环：当 `complexCheck()` 走真实 Qwen 成功调用时，`modelCallLog` 中应保留清晰的 Qwen 接入标识，便于从日志层直接观察到真实模型提供方。

本轮不处理：

- Mock 降级治理增强。
- 复核反馈回流决策追踪。
- Phase 5 文档交付收口。

## 方案

### 1. 观测点选择

不改动 `QwenModelGateway` 的对外协议，不新增额外数据库表，也不改动 `decisionTrace` 结构。最小方案是在模型成功日志写入时，把模型提供方信息拼入已存在的可存储字段中，让测试与排查都能直接观察到 `qwen`。

### 2. 变更边界

- 优先修改 `ExceptionAnalysisOrchestratorImpl`，因为它掌握当前使用的 `ModelGateway` 实例与成功日志写入时机。
- 若需要最小暴露提供方标识，则在 `ModelGateway` 接口上补一个极小的默认能力，用于返回 provider 标识；现有实现保持兼容。
- 不改控制器契约，不改前端，不改数据库结构。

### 3. 测试策略

采用 TDD：

1. 先在后端集成测试中补一条失败用例。
2. 触发 `complexCheck()` 成功路径。
3. 断言 `modelCallLog` 成功记录中包含 `qwen` 标识。
4. 再做最小实现让测试通过。

## 风险与约束

- 只允许影响本轮最小观测字段，不能破坏现有成功日志、失败日志、异常生成、决策追踪行为。
- 若现有测试基线对日志内容做了严格断言，需要同步以最小方式更新。
- 不依赖真实外网调用；测试通过 mock `ModelGateway` 成功返回即可验证日志观测闭环。
