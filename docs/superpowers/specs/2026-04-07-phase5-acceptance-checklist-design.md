# Phase 5 最终验收清单单文档设计

## 背景

`Phase 5 - 文档、部署、联调、验收闭环` 当前已完成第一包：

- 后端单仓库启动文档：`README.md`
- 前端单仓库启动文档：`README.md`
- 本地联调手册：`docs/integration/phase5-local-runbook.md`

当前剩余缺口不再是“怎么启动”或“怎么联调”，而是“怎么最终验收并留存证据”。根据状态文档约束，第二包需要把以下内容收口为稳定交付物：

- 最终验收步骤
- 证据清单
- 截图要求

本轮选择单文档收口，不拆附录。

## 目标

新增一份主文档：

`D:\Graduation project\backend\.worktrees\phase5-docs-runbook\docs\integration\phase5-acceptance-checklist.md`

使新会话在只读取最小必要文件的前提下，能够：

1. 按顺序执行最终验收，而不是自由发挥。
2. 明确知道每一步需要保留什么证据。
3. 明确知道每一步至少需要什么截图。
4. 区分“必验项”和“可选增强项”，避免把验收范围无限扩大。
5. 形成是否可以把 `Phase 5` 标记为完成的明确判定口径。

## 范围

### 本轮包含

- 新增一份单文档验收清单：`docs/integration/phase5-acceptance-checklist.md`
- 回写 `D:\Graduation project\继续执行文档-完整系统.md`，把第二包状态、最小读取清单、验证口径与下一步收口结果写清楚

### 本轮不包含

- 再次重写后端 README、前端 README、联调手册
- 新增脚本、自动截图、自动收证工具
- 修改前后端业务代码
- 把可选增强链路强行升级为首个完成门槛

## 单文档组织方式

推荐按真实验收顺序组织，而不是按证据类型拆块。文档结构固定为 6 段：

### 1. 文档用途与适用范围

说明这是一份“最终验收与留证”文档，不替代：

- 后端 README
- 前端 README
- `phase5-local-runbook.md`

执行顺序应是：

1. 先按 README 准备环境
2. 再按联调手册启动与联调
3. 最后按本清单做最终验收和留证

### 2. 验收前准备

这部分只列最终验收前必须明确的前提，不重复解释启动细节：

- 当前后端 worktree：`backend/.worktrees/phase5-docs-runbook`
- 当前前端 worktree：`frontend/.worktrees/phase5-docs-runbook`
- 后端、前端、联调手册都已处于当前 worktree 的最新文档版本
- MySQL、Redis、本地配置、前端代理已经按首包文档准备完成
- 是否启用可选增强链路（实时新打卡、地图、复杂异常、多地点异常）

### 3. 启动验收

验证“服务可启动、测试可执行、构建可完成”。

必验项只包含：

- 后端最小测试通过
- 后端编译通过
- 前端测试通过
- 前端构建通过

这部分不要求重复解释命令来源，只要求给出：

- 操作
- 预期
- 证据
- 失败时记录

### 4. 联调验收

验证“主链可走通并能留证”。

最小基线只覆盖：

- 登录
- 异常查询
- 预警查询
- 复核提交与状态流转
- 统计查询

该部分继续保持 `seed` 数据优先策略，不把“实时新打卡”写成稳定基线。实时新打卡、地图、复杂异常、多地点异常统一放进“可选增强验收”。

### 5. 证据与截图要求

这部分是第二包的核心。每类验收项都要固定列出：

- 至少保留哪一段命令输出
- 至少保留哪个页面截图
- 至少核对哪个状态变化或数据结果
- 如果失败，必须记录什么上下文

不要求自动采集，也不伪造仓库中不存在的截图模板。

### 6. 最终通过判定与遗留问题记录

这部分明确回答：什么情况下可以把 `Phase 5` 标记为完成。

原则：

- 必验项全部通过，才允许写“Phase 5 完成”
- 可选增强项未验证时，只能写“可选项未验 + 原因”，不能阻止首个完成判定
- 非阻塞问题必须以“已知问题/遗留问题”方式留档，不能静默忽略

## 验收口径

### 必验项

第二包只把以下 4 类定义为“必验”：

1. 启动证据
   - 后端最小测试通过
   - 后端编译通过
   - 前端测试通过
   - 前端构建通过
2. 主链证据
   - 登录成功
   - 异常可查询
   - 预警可查询
   - 复核提交成功并带来状态变化
   - 统计结果可查询
3. 文档一致性证据
   - README、联调手册、验收清单三者的路径、命令、端口、验证口径一致
4. 问题记录
   - 非阻塞项必须明确记录，不得直接忽略

### 可选增强项

以下内容允许写成“可选增强验收”：

- 实时新打卡链路
- 地图展示
- 复杂异常分析
- 多地点异常
- 导出类页面演示截图

原因：这些能力依赖额外配置或额外样本准备，适合增强验证，但不应把第二包拖成无限扩展任务。

## 证据模板原则

为了让单文档可直接执行，每个验收项统一采用 4 行模板：

- 操作
- 预期
- 证据
- 失败时记录

这样文档既能指导执行，也能指导留证，不会退化成只会打勾的空清单。

## 风险与边界

- 不伪造截图模板、录屏能力或自动采证能力
- 不伪造真实密钥、账号或外部平台权限
- 若某项依赖外部能力不可验证，写成“可选增强未验 + 原因”
- 不回头大改第一包文档，除非发现和第二包直接冲突的口径错误

## 验证策略

本轮仍是文档交付，因此验证重点是“文档可执行性与一致性”，不是新增业务代码测试。至少要完成：

1. 校对第二包文档中的命令、端口、页面入口与第一包三份文档保持一致。
2. 校对“必验项/可选增强项”的边界与当前真实系统能力一致。
3. 回写 `继续执行文档-完整系统.md`，把第二包的状态、最小读取清单、最小验证与下一步收口结果写清楚。

- 由于第二包主要新增验收文档与状态回写，默认不要求新建前端文件；若需要调整前端 README，再在现有前端 worktree `frontend/.worktrees/phase5-docs-runbook` 内最小修改
