# BE-01 用户管理请求解析异常 400 语义设计

## 1. 目标

在不改变现有用户管理成功语义、鉴权语义和业务校验语义的前提下，把畸形 JSON 或请求体解析失败这类输入错误从当前的通用 `500` 调整为业务 `code=400`，与现有 API 文档中的“请求参数错误”语义保持一致。

## 2. 范围

- 纳入范围：`POST /api/user/add`、`PUT /api/user/update` 这类基于 `@RequestBody` 的用户接口在收到畸形 JSON 时的返回语义。
- 纳入范围：全局异常处理器对请求体解析异常的统一收口。
- 不纳入范围：数据库结构、业务校验规则、成功返回结构、鉴权规则、并发问题。
- 暂不扩展：其他未在本轮验证范围内的参数异常类型，不做额外语义调整。

## 3. 设计方案

### 3.1 异常处理

在 `GlobalExceptionHandler` 中新增针对 Spring 请求体解析异常的处理分支，优先覆盖 `HttpMessageNotReadableException`。

处理规则：

- HTTP 状态仍由现有接口风格保持为 `200`
- 返回体统一为业务失败结构
- `code` 固定为 `400`
- `message` 使用现有通用语义 `请求参数错误`

这样可以避免把底层 JSON 解析细节直接暴露给前端，同时与当前项目返回风格保持一致。

### 3.2 测试策略

采用 TDD：

1. 先在 `UserManagementIntegrationTest` 为新增用户接口补畸形 JSON 用例，验证当前失败现象。
2. 再做最小实现让该测试转绿。
3. 运行最小相关回归，确认没有破坏既有用户管理行为。

本轮先锁定一个用户接口作为代表场景，避免扩大改动面；由于异常处理器是全局生效，后续如需要再补更多接口覆盖。

## 4. 文档影响

- `docs/test/测试用例文档.md` 补充或细化本轮新增的异常输入验证范围。
- `docs/api/API接口设计文档.md` 当前已经声明 `400` 为请求参数错误，本轮若无需补充更细粒度描述，可不做额外结构调整。

## 5. 风险与取舍

- 本轮只修复请求体解析失败，不一并扩展所有 Spring 参数绑定异常，目的是保持最小必要改动。
- 若后续需要把查询参数类型错误、缺失参数等也统一到 `400`，可以在同一异常处理器中继续增补，但不属于本轮范围。
