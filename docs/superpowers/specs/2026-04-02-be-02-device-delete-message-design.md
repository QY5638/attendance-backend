# BE-02 设备删除失败提示优化设计

## 1. 背景

当前 `BE-02` 设备基础资料模块已经实现以下删除规则：

- 调用 `DELETE /api/device/{deviceId}` 时，若设备不存在，返回业务错误
- 若设备已被 `attendanceRecord.deviceId` 引用，则禁止删除

现有规则本身没有问题，但失败提示只表达“不能删除”，没有直接告诉调用方下一步应该如何处理。

## 2. 目标

本次只优化“设备已被引用时”的失败提示，让接口在保持现有业务规则不变的前提下，明确引导调用方改为停用设备。

完成后，删除已被打卡记录引用的设备时，接口仍然失败，但错误消息升级为更明确的引导语。

## 3. 非目标

本次不包含以下内容：

1. 不修改删除业务规则本身
2. 不把删除请求自动转成停用操作
3. 不新增删除前预检查接口
4. 不新增响应字段或调整统一响应结构
5. 不调整前端页面交互

## 4. 方案

### 4.1 保持现有删除规则

- 设备不存在：继续返回业务错误
- 设备已被 `attendanceRecord` 引用：继续禁止删除
- 设备未被引用：继续允许删除

### 4.2 仅优化业务错误文案

将“设备已被打卡记录引用”的失败消息从当前的：

`设备已关联打卡记录，不能删除`

调整为更明确的引导语：

`设备已关联打卡记录，不能删除，请先停用设备`

该方案不改变：

- HTTP 状态码
- 统一响应体结构
- 业务错误码语义

## 5. 影响范围

本次改动只涉及以下 3 类内容：

1. `device` 删除校验逻辑中的提示文案
2. `DeviceManagementIntegrationTest` 中删除已引用设备场景的断言
3. 设备 API 与测试文档中的对应描述

## 6. 验证

最小验证：

1. 运行 `DeviceManagementIntegrationTest`
2. 确认删除已引用设备时，仍返回失败
3. 确认返回消息为 `设备已关联打卡记录，不能删除，请先停用设备`

如本次改动波及公共断言或文档同步范围超出预期，再补充执行 `mvn test` 做全量回归。

## 7. 风险与取舍

- 优点：改动小、兼容性好、不会扩大 `BE-02` 范围
- 风险：前端仍需通过 `message` 文本理解“建议停用”的下一步动作
- 结论：在当前阶段，这个取舍是可接受的，优先保证规则稳定和提交边界清晰
